# TabVault Pro Co-Founder Roadmap

> 目标：把 TabVault Pro 从“标签页管理插件”收紧为“工作会话保险箱”，先修复信任与定位，再推进增长与付费。

---

## 1. 战略定位

### 一句话卖点

`TabVault Pro 是给重度浏览器用户的工作会话保险箱。`

### 目标用户

- 开发者：长期同时打开文档、PR、Issue、监控、后台与教程。
- 研究型用户：长期打开论文、资料、论坛、竞品、视频与长文。
- 内容作者：长期打开选题、草稿、素材、引用、后台与参考链接。
- 招聘、销售、投资人：长期打开候选人、CRM、公司页、邮件、表格与 Notion。

### 核心价值

- 保存当前工作上下文，而不是只“收纳标签页”。
- 之后能快速找回，而不是重新搜一遍。
- 能在不同设备上延续工作，而不是从头开始。

### 对外关键词

- Save the session
- Find it later
- Restore anywhere

### 不再主打的叙事

- 主题很多
- 功能很多
- OneTab 替代品
- 超前承诺的“实时同步”

---

## 2. 北极星指标

- 每周至少保存 3 次会话的活跃用户数
- 7 日内至少恢复过 1 次会话的用户占比
- 首次安装后 24 小时内完成“保存一次 + 恢复一次”的转化率
- OneTab 导入用户的次周留存
- 登录用户中至少完成 1 次成功同步的占比

---

## 3. 6 周 Roadmap

### Week 1：信任修复

目标：先把产品“说真话、跑得稳、看起来可信”。

- 删除开发残留、可疑调试代码和本地上报逻辑。
- 统一版本号、README、页脚、安装文案、商店文案。
- 把所有“实时同步”表述改为当前真实能力。
- 建立最小回归清单：保存全部、恢复全部、OneTab 导入、登录、手动同步。
- 梳理隐私与加密表述，避免过度承诺。

### Week 2：从标签页升级为会话

目标：让用户感知到这是“工作上下文恢复工具”，不是普通标签收纳盒。

- 保存时支持默认命名规则。
- 支持快速重命名会话。
- 恢复时优先恢复到新窗口。
- 恢复完成后给出明确反馈：恢复数量、是否保留 pinned、原会话状态。
- 把“标签组”对外文案逐步替换成“会话”。

### Week 3：把找回做强

目标：让用户愿意回来，而不只是保存完就忘。

- 搜索结果优先展示会话。
- 增加按域名、时间、关键词过滤。
- 增加“最近保存”“最近恢复”“最近访问来源”。
- 增加会话备注字段。
- 优化空状态与无结果状态，强化可找回心智。

### Week 4：把同步做成可依赖能力

目标：同步不是“高级选项”，而是高信任功能。

- 明确同步策略、冲突策略、覆盖策略。
- 增加同步状态页或同步说明区。
- 把同步成功、失败、冲突原因讲清楚。
- 如果暂时无法恢复自动同步，就把手动同步做到足够稳。
- 只有在成功率与冲突处理足够明确后，再考虑恢复自动同步。

### Week 5：首日留存和增长

目标：安装后尽快让用户体验到“保存工作现场”的核心价值。

- 重写 onboarding，只保留保存、搜索、恢复三条主路径。
- 强化 OneTab 导入入口，把迁移做成增长渠道。
- 优化首次保存、首次恢复的引导反馈。
- 设计一套面向重度用户的商店截图与文案。
- 收集最小产品事件埋点，观察首日转化漏斗。

### Week 6：试水付费

目标：验证“跨设备连续性”和“安心感”是否有人愿意付费。

- 设计免费版与 Pro 版边界。
- 免费版保留本地核心价值，不阉割基本体验。
- Pro 主打跨设备同步、历史版本、误删恢复、高级恢复。
- 输出 Chrome Store 上架素材与 FAQ。
- 找 20 个重度用户做访谈，只验证两个问题：
  - 他们会在什么场景下保存。
  - 他们会不会回来恢复。

## 4. 产品包装建议

### 免费版

- 本地无限保存与恢复
- 基础搜索
- OneTab 导入导出
- 基础整理与重命名
- 基础设置

### Pro 版

- 跨设备同步
- 自动云备份
- 历史版本与误删恢复
- 恢复策略增强
- 高级搜索过滤
- 会话备注与收藏
- 后续可增加 AI 自动命名与摘要

### 建议定价

- $3.99 / 月
- $29 / 年
- 早期用户价可测试 $19 / 年

### 定价原则

- 不对“本地保存”收费。
- 对“安心感”和“跨设备连续性”收费。

## 5. 当前项目的关键风险

### 风险 1：产品承诺和实现脱节

- 文案与 onboarding 仍在强调“实时同步”。
- 当前实现以手动同步为主。
- 这会直接损伤用户信任与留存。

### 风险 2：可信度细节不一致

- README、包版本、页脚版本不一致。
- README 联系方式和仓库地址仍是占位内容。
- 这会降低商店转化与外部合作信心。

### 风险 3：隐私与安全叙事过重

- “端到端加密”属于高承诺表述。
- 若底层模型和密钥方案不足以支撑严格意义上的 E2E，会形成品牌风险。

### 风险 4：工程复杂度已经先于产品验证

- 同步、设置、主题、拖拽、迁移、性能测试都已存在。
- 但产品最重要的“保存一次后会不会回来恢复”还没有被明确验证。

---

## 6. 执行原则

### 原则 1：先修信任，再做增长

- 任何会影响“数据是不是安全、能不能找回”的问题都优先于新功能。

### 原则 2：先做核心闭环，再做周边能力

- 核心闭环是：保存 -> 找到 -> 恢复 -> 信任增强。

### 原则 3：所有新工作必须服务于重度用户

- 对重度用户没有明显价值的功能，先不做。

### 原则 4：不再对外承诺未完成能力

- 特别是同步、隐私、安全、跨设备一致性。

---

## 7. 第一阶段执行计划

### Phase A：P0 信任修复

目标：1 周内把产品收敛到“可以放心继续投入”的状态。

- 移除所有开发/调试残留与可疑日志。
- 统一版本号来源，页面、包、文档、清单全部同源。
- 修正 README、页脚、onboarding、设置文案。
- 统一同步叙事为“手动同步”或“当前同步模式”。
- 审核隐私相关表述。

交付标准：

- 外部文案不再承诺不存在的功能。
- 不再存在明显的本地调试上报逻辑。
- 用户从 README、安装页、产品内看到的是同一版本和同一能力描述。

### Phase B：P1 会话化升级

目标：把用户理解从“标签组”切换到“工作会话”。

- 明确“会话”命名与文案。
- 设计默认命名规则。
- 优化保存后的反馈与恢复路径。
- 增加备注与最近恢复入口。

交付标准：

- 新用户能快速理解“这是保存工作现场的工具”。
- 保存后 24 小时内的恢复率可开始观测。

### Phase C：P2 同步与付费验证

目标：在信任稳定后，再增强同步和商业化。

- 梳理同步状态、冲突、覆盖与恢复说明。
- 观察登录到首次同步转化。
- 设计 Pro 边界并准备试水。

交付标准：

- 用户知道同步做了什么、没做什么、失败了怎么办。
- 有清晰的免费版与付费版分层逻辑。

## 8. 首批执行任务清单

### Immediate Tasks

- 删除 `TabManager` 中的本地调试上报与冗余 debug 代码。
- 用运行时版本号替换硬编码页脚版本。
- 修正 onboarding 中的“实时同步”文案。
- 更新 README 到当前版本与真实能力。
- 清理 README 中的占位链接与邮箱。
- 建立一份最小自测 checklist。

### 验证方式

- 本地手动保存全部标签后，数据完整，页面提示正确。
- 从已保存会话恢复标签后，行为和提示一致。
- 未登录状态下不会误导用户认为云同步已开启。
- 所有入口看到的同步表述一致。

---

## 9. 决策口径

后续所有产品和工程决策，默认用这三个问题判断是否值得做：

1. 它是否让用户更放心地保存当前工作现场？
2. 它是否让用户更容易找回之前保存的内容？
3. 它是否让用户更愿意在不同设备上继续之前的工作？

如果三个问题都答不上来，默认不做。
